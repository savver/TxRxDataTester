# RxDataTester (v.1.0)

Приемник тестового счетчика для последовательного COM-порта.

Проект рассчитан на **Qt 5.12**, собирается через **qmake** и использует модуль
`serialport`.

## Состав проекта

```text
RxDataTester.pro
main.cpp
mainwindow.h
mainwindow.cpp
mainwindow.ui
rxworker.h
rxworker.cpp
```

## Сборка

Открыть `RxDataTester.pro` в Qt Creator с комплектом Qt 5.12 либо выполнить:

```text
qmake RxDataTester.pro
make
```

Для MSVC после `qmake` используется обычная сборка выбранного комплекта Qt.

## Архитектура потоков

Программа разделена на два потока.

```text
Главный GUI-поток
├─ MainWindow и все виджеты
├─ кнопки OPEN / CLOSE / START / STOP
├─ перечисление QSerialPortInfo раз в секунду
├─ цветное окно EVENTS
├─ текстовый лог и flush()
└─ QSettings

Рабочий RX-поток
├─ RxWorker
├─ QSerialPort
├─ обработка readyRead()
├─ входной QByteArray с неполным хвостом
├─ разбор little-endian счетчиков
├─ сравнение с ожидаемым значением
├─ QTimer Statistics
├─ QTimer контроля состояния порта
└─ расчет скорости и двадцатисекундной статистики
```

GUI не читает `QSerialPort` напрямую. Все команды передаются в `RxWorker` через
`Qt::QueuedConnection`, а в обратную сторону приходят готовые события и снимки
Statistics. Перерисовка или сворачивание окна не останавливают чтение порта.

## Settings

Поддерживаются:

```text
Baud:   115200, 230400, 460800, 921600, 1500000
Data:   8 bits
Parity: NONE, EVEN, ODD
Stops:  1 или 2
Flow:   NONE
```

Пока порт закрыт, список COM-портов обновляется раз в секунду. Появление и
исчезновение устройств регистрируются в EVENTS и лог-файле. При открытом порте
контролируется наличие именно используемого устройства.

`OPEN` открывает порт в режиме `QIODevice::ReadOnly`. После открытия Port, Baud,
Parity и stops блокируются до `CLOSE`.

## Pattern

```text
counter, bits : 8, 16, 32 или 64
block, bytes  : информационный размер блока передатчика
init value    : десятичное число или HEX с префиксом 0x
Period, ms    : информационный период блоков передатчика
block len, us : расчетное время блока по параметрам UART
bo            : LE, little-endian
```

`block, bytes` автоматически округляется вверх до кратности размеру счетчика.
Поля `block, bytes` и `Period, ms` сохранены для согласования настроек с
TxDataTester, расчета `block len, us` и журнала. **Границы блока для приема не
используются.**

## Алгоритм проверки счетчика

После `START` программа отбрасывает данные, накопленные до запуска теста, и
ожидает новые байты. Поток разбирается непрерывно только по размеру счетчика:

```text
8 bits  -> 1 байт
16 bits -> 2 байта
32 bits -> 4 байта
64 bits -> 8 байт
```

Вызов `readyRead()` может принести любое число байт. Неполные 1...7 байт
сохраняются во внутреннем буфере до следующего прихода данных. Каждый полный
счетчик собирается вручную в little-endian, поэтому алгоритм не зависит от
порядка байт процессора.

Пример для `counter=32 bits`, `init value=10`:

```text
принято 10 -> counter, ok + 1; далее ожидается 11
принято 11 -> counter, ok + 1; далее ожидается 12
...
```

При несовпадении увеличивается `counter, err` только на один, независимо от
разницы чисел. Новым ожидаемым значением становится принятое значение плюс один:

```text
ожидалось 120
принято 124
counter, err + 1
следующее ожидаемое 125
```

В EVENTS и лог записывается красная строка:

```text
19:44:06.361 - ошибка счетчика: ожидалось 120, принято 124, в следующий раз ожидается 125
```

После максимального значения счетчик переполняется и продолжается с нуля.
Алгоритм предполагает сохранение байтового выравнивания потока. Если потерян не
целый счетчик, а отдельный байт внутри значения, автоматический поиск новой
границы в версии 1.0 не выполняется.

## Statistics

Раз в секунду обновляются:

```text
start:        локальное время нажатия START
 time:        длительность теста без сброса после 24 часов
rx, bytes:    все байты, принятые после START
curr_count:   последнее полностью принятое значение счетчика
speed, Kb/s:  фактическая скорость за реальный измеренный интервал
counter, ok:  количество совпавших значений
counter, err: количество несовпавших значений
```

Скорость вычисляется по сигналам приема и монотонному времени:

```text
speed_Kb_s = delta_rx_bytes * 8 / delta_time_ms
```

Биты в миллисекунду численно равны десятичным килобитам в секунду.

## EVENTS и цвета

Формат обычной строки:

```text
ЧЧ:ММ:СС.мс - событие
```

Цвета:

```text
зеленый — непосредственное нажатие OPEN, CLOSE, START или STOP
красный  — ошибки порта, чтения, Pattern и несовпадения счетчика
черный   — запуск программы, сервисная информация, открытие/закрытие порта
```

Окно использует моноширинный шрифт, автоматическую прокрутку и ограничение в
10 000 видимых строк. Текстовый файл ограничение строк не имеет.

## Лог-файлы

При каждом запуске рядом с исполняемым файлом создается каталог:

```text
logs
```

Имя нового файла:

```text
rxdatatester_log__2026-07-22__19-44-06-361.txt
```

Все строки EVENTS дублируются в UTF-8 файл. Дополнительно раз в 20 секунд только
в файл записывается строка Statistics, например:

```text
19:44:26.361, time=00:00:20, rx_bytes=1782656, delta_rx_bytes=1782656, curr_counter=445689, counter_ok=445690, delta_counter_ok=445690, counter_err=0, delta_counter_err=0, min_speed=522, avrg_speed=544.18, max_speed=621
```

`avrg_speed` рассчитывается по точному фактическому интервалу, а `min_speed` и
`max_speed` выбираются среди секундных значений, показанных в GUI.

## Сохранение настроек

При нормальном закрытии через `QSettings` сохраняются:

```text
геометрия окна
выбранный COM-порт
Baud, Parity, stops
counter, bits
block, bytes
init value, включая формат 0x
Period, ms
```

При следующем запуске значения восстанавливаются и повторно проверяются.
