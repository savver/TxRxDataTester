# TxDataTester (v.1.4)

Проект рассчитан на **Qt 5.12**, модуль **Qt Serial Port** и сборку через
**qmake**. CMake не используется. Вкладка UDP пока оставлена пустой.

## Сборка

Открыть `TxDataTester.pro` в Qt Creator либо выполнить в консоли комплекта Qt:

```text
qmake TxDataTester.pro
mingw32-make
```

Для комплекта MSVC вместо `mingw32-make` используется соответствующая команда
`nmake` или `jom`.

## Изменения версии 1.4

Передача данных отделена от интерфейса в самостоятельный рабочий поток.
Архитектура программы теперь выглядит так:

```text
Главный GUI-поток
├─ MainWindow и все виджеты
├─ кнопки OPEN / CLOSE / START / STOP / SINGLE
├─ получение списка QSerialPortInfo раз в секунду
├─ цветное окно EVENTS
├─ запись и flush текстового лог-файла
└─ QSettings

Рабочий TX-поток QThread
├─ TxWorker
├─ QSerialPort
├─ QTimer формирования блоков
├─ QTimer Statistics
├─ QTimer контроля неожиданного закрытия порта
├─ заполнение QByteArray счетчиком
├─ вызовы QSerialPort::write()
├─ обработка bytesWritten и errorOccurred
├─ мягкий STOP с допередачей остатка
└─ расчет скорости и двадцатисекундной Statistics
```

`QSerialPort` и все его таймеры создаются методом `TxWorker::initialize()` уже
после переноса `TxWorker` в `QThread`. Поэтому они с самого начала принадлежат
рабочему потоку. GUI не читает `isOpen()`, `bytesToWrite()` и другие свойства
`QSerialPort` напрямую.

Команды передаются только queued-сигналами:

```text
GUI -> openPortRequested(...)
GUI -> closePortRequested()
GUI -> startContinuousRequested(...)
GUI -> stopContinuousRequested()
GUI -> singleRequested(...)
GUI -> externalPortLossDetected(...)
```

Результаты возвращаются queued-сигналами:

```text
TX -> portStateChanged(...)
TX -> transmissionStateChanged(...)
TX -> statisticsUpdated(...)
TX -> eventGenerated(...)
TX -> periodicLogLineReady(...)
TX -> stopButtonAccepted(...)
```

Практический результат:

- перерисовка, перемещение и изменение размера окна больше не задерживают
  формирование очередного блока;
- сворачивание окна не влияет на рабочий таймер передачи;
- медленный `flush()` лог-файла не останавливает QSerialPort;
- проверка списка COM-портов в GUI не прерывает формирование данных;
- при `Period = 0` рабочий поток самостоятельно поддерживает выходную очередь;
- расчет секундной и двадцатисекундной Statistics не зависит от перерисовки GUI.

Windows и `QTimer` по-прежнему не являются системой жесткого реального времени:
системная нагрузка может немного изменять интервалы 0–1 мс. Отдельный поток
устраняет влияние GUI, но не заменяет аппаратный таймер.

События рабочего потока получают метку `ЧЧ:ММ:СС.мс` непосредственно в момент
операции. Даже если GUI временно занят, в EVENTS и лог попадает время события, а
не более позднее время его отображения.

При завершении программы GUI вызывает `TxWorker::shutdown()` через
`Qt::BlockingQueuedConnection`, дожидается закрытия порта и остановки рабочего
потока, принимает итоговые события и только после этого закрывает лог-файл.

## Основные возможности

- Заголовок окна `TxDataTester (v.1.4)`.
- Надпись `Tranceiver`: 12 pt, нежирный шрифт.
- Вкладки COM и UDP; вкладка UDP пока пустая.
- Реальное открытие и закрытие COM-порта через `QSerialPort`.
- Настройки порта: 8 data bits, выбранные baud/parity/stops и
  `NoFlowControl`.
- При открытом порте нельзя менять Port, Baud, Parity и stops.
- Каждую секунду программа получает новый список портов. При закрытом порте в
  EVENTS регистрируются появившиеся и исчезнувшие устройства. При открытом
  порте контролируется наличие используемого устройства.
- В рабочем потоке дополнительно раз в секунду проверяется, что логически
  открытый `QSerialPort` не закрылся неожиданно без отдельного события списка.
- Все ошибки отображаются красным, обычные и сервисные события — черным.
  Зеленым выделяется только непосредственное нажатие START, STOP или SINGLE.
- EVENTS имеет моноширинный шрифт, цветные строки, автоматическую прокрутку и
  ограничение истории в 10 000 строк.
- Каждая событийная строка имеет метку `ЧЧ:ММ:СС.мс`.
- При каждом запуске рядом с исполняемым файлом создается каталог `logs` и новый
  UTF-8 файл вида:

```text
txdatatester_log__2026-07-22__19-43-46-127.txt
```

- Через `QSettings` сохраняются геометрия окна, настройки COM и поля Pattern.
- Поля `block, bytes` и `Period, ms` принимают только цифры `0...9`.
- `block, bytes` округляется вверх до значения, кратного размеру счетчика.
- `init value` принимает десятичное число или HEX с префиксом `0x`.
- Время Statistics не сбрасывается после 24 часов.

## Формирование блока

Счетчик беззнаковый, разрядность выбирается из 8, 16, 32 или 64 бит. Значения
помещаются в блок последовательно в порядке **little-endian**. После достижения
максимального значения счетчик переполняется и продолжает работу с нуля.
Между соседними блоками START счетчик не сбрасывается.

Для `block = 64 bytes` и `counter = 16 bits` в блок помещается:

```text
64 / 2 = 32 значения
```

При `init = 0` первый блок содержит значения `0...31`, второй — `32...63`,
третий — `64...95`. Число `1` передается байтами `01 00`.

## Period, ms

- `Period > 0` — рабочий таймер пытается формировать очередной блок через
  заданное число миллисекунд.
- `Period = 0` — заданная программная пауза отсутствует. Однократный нулевой
  таймер пополняет очередь, а после ее заполнения ожидает `bytesWritten`.
- При заполненном окне очереди новый блок не создается.
- Окно равно максимуму из 4096 байт и четырех активных блоков.

Период задает моменты постановки блоков в очередь, а не гарантированную
аппаратную паузу после последнего стоп-бита предыдущего блока.

## block len, us

Метка пересчитывается автоматически при изменении `block, bytes`, Baud, Parity
или stops. Для одного байта учитываются:

```text
1 start bit + 8 data bits + parity bit при EVEN/ODD + 1 или 2 stop bits
```

Расчет:

```text
block_len_us = ceil(block_bytes * bits_per_byte * 1 000 000 / baud)
```

Например, для 128 байт, 921600 baud, `NONE`, 2 stop bits:

```text
block len, us => 1528
```

Это теоретическое время передачи по линии без задержек драйвера, USB и ОС.

## Поведение STOP

1. Команда STOP queued-сигналом попадает в рабочий поток.
2. Рабочий таймер формирования новых блоков немедленно останавливается.
3. TxWorker считывает `bytesToWrite()` и возвращает размер хвоста GUI.
4. GUI выводит зеленую строку нажатия:

```text
02:08:01.123 - нажата кнопка STOP; формирование новых блоков остановлено; в очереди 4096 байт
```

5. `clear(Output)` не вызывается.
6. Уже поставленные байты продолжают передаваться.
7. `bytesWritten`, Statistics и двадцатисекундный лог продолжают учитываться.
8. После `bytesToWrite() == 0` рабочий поток выводит обычную черную строку:

```text
02:08:01.171 - STOP завершен; остаток выходной очереди передан; всего передано 12345678 байт
```

Явный CLOSE, завершение программы или потеря порта могут принудительно удалить
остаток, поскольку в этих случаях устройство требуется закрыть.

## EVENTS

Пример:

```text
02:30:43.004 - нажата кнопка START
02:30:43.005 - START: циклическая передача запущена; Pattern: counter=8 bits; init=0; block=128 bytes; period=2 ms; values/block=128; bo=LE
02:36:22.100 - нажата кнопка SINGLE
02:36:22.101 - SINGLE: один блок поставлен на передачу; Pattern: counter=8 bits; init=0; block=128 bytes; period=3 ms; values/block=128; bo=LE
02:36:22.345 - SINGLE: передача блока завершена, передано 128 байт
```

Обычные строки EVENTS полностью дублируются в текстовый лог.

## Statistics

Расчет выполняется в рабочем потоке:

- `start:` — локальное время запуска START или SINGLE;
- `time:` — длительность монотонного `QElapsedTimer`;
- `tx, bytes` — сумма подтверждений `QSerialPort::bytesWritten`;
- `curr_count` — следующее значение счетчика после сформированного блока;
- `speed, Kb/s` — скорость за фактический секундный интервал:

```text
speed = delta_bytes * 8 / delta_time_ms
```

Результаты передаются GUI готовым сигналом `statisticsUpdated(...)`.

## Дополнительная строка текстового лога

Во время START раз в 20 секунд TxWorker формирует строку:

```text
19:44:06.361, time=00:00:20, tx_bytes=1360450, delta_tx_bytes=1360450, curr_counter=445689, min_speed=522, avrg_speed=544.18, max_speed=621
```

Она не выводится в EVENTS. GUI-поток только записывает готовую строку в файл.

- `delta_tx_bytes` — разность общего объема между соседними строками;
- `min_speed` и `max_speed` — минимум и максимум секундных скоростей за окно;
- `avrg_speed` — `delta_tx_bytes * 8 / фактический_интервал_мс`.
